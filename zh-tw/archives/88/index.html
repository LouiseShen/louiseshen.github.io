真·靜態化Typecho，讓訪問速度真正起飛

  • 釋出時間：2020-02-17 19:19:00
  • 4713字數
  • 分類： 伺服器
  1. 首頁
  2. 正文

對於只有1核心的伺服器，跑上mysql簡直可以感覺到緩慢速度了。

Head Pic: original, cat ears, beast ears / 雫♡zz - pixiv

當初從最初優化時的發現可互動前耗時爆紅開始，我便意識到了這個問題，這種延遲是任何一個在意速度的訪客無法忍受的。爲了拋棄日漸大頭的動態內容渲染耗時，沒有什麼優化外掛我沒有試過，也沒有什麼方法我沒有求助過，更沒有什麼免費的CDN我沒有用過。我為cloudflare自選了節點，因為從我聽說地理位置與訪問速度有直接聯絡。我......（寫不下去了哈哈哈哈）

迴歸主題，既然加速動態內容（指透過查詢資料庫來實現的）在1核主機和不花錢的情況下幾乎很難做到，那就乾脆不要mysql和php了，咱們直接上html靜態頁面。

但是原問題沒有得到妥善處理，新問題又不斷出現，直接靜態化的話我們總不可能重新再一個個頁面手動生成，而且部分依靠動態內容才能實現的東西基本上沒指望了。

而我這邊花了半天時間基本上解決了上述問題，並且成功實現了靜態化，在Google的PageSpeed Insights桌面測試中也順利拿下了92分，移動端因為js過複雜只有62分。

優勢與不足

透過靜態化，會有這樣的好處：

  • 理論上CC攻擊相比於用mysql的1核主機更難攻陷（nginx訪問html資源消耗可以說非常低了）
  • CDN可以直接Cache Everything
  • 頁面渲染速度會快很多。（少掉了php解析和資料庫環節）
  • 可以部署在任何支援靜態頁面託管的地方（比如拿github pages作備用站，源站被ddos直接切換解析）
  • 等等

但是壞處也是極其明顯的：

  • 評論和瀏覽量統計需要依靠其他服務
  • 釋出文章不是很方便（簡直不方便了）
  • 更新一篇文章，與之關聯都要變動
  • 一些需要PHP的外掛也會失效（比如追番外掛，雖然我也把它靜態化了）
  • 等等

所以基本上靜態化就是一把雙刃劍，全憑諸位操作。

評論沒了，沒關係，反正也很少有人評論。

瀏覽次數沒了，沒關係，瀏覽人數難道自己心裏沒底嘛。

所以我就靜態化了，並且有了這篇水文。

生成頁面

因為是靜態頁面，所以我們新建一個資料夾，然後把主題中的靜態資源複製到這個資料夾，

這裏以web和Handsome主題為例：

目前的目錄結構：

web/
|_usr/

usr是此主題全部css和js等等靜態資源，所以我們直接拷貝過來，並且刪除其中的php檔案（不要洩露主題）。

然後就是文章了，首先curl命令可以訪問https/http，並得到網頁原始碼：

curl https://4o5.xyz > index.html

這樣就是把我的主頁儲存到當前目錄下的index.html，然後我們把它放到web目錄中。

目前的目錄結構：

web/
|_usr/
|_index.html

現在開啟瀏覽器就可以訪問到你的網站了，當然有些資源不能載入，因為你是本地檔案的方式開啟的。

（這裏強烈建議在本地跑一個nginx之類的網頁伺服器，並把你的域名hosts到127.0.0.1觀察效果）

這樣就解決了網頁的問題，~~接下來所有的文章頁面分類等等按這個模式一步一步來就好啦！~~

咳咳，要是每個都要手動新增豈不是太痛苦了，所以我寫了個java程式：

import java.io.BufferedReader;
import java.io.File;
import java.io.InputStreamReader;
import java.util.Scanner;

public class Generator {
	public static void main(String[] args) {
		Scanner in = new Scanner(new BufferedReader(new InputStreamReader(System.in)));
		String s,backup,dir;
		String[] delta = new String[1024];
		
		boolean isWin = System.getProperty("os.name").indexOf("Windows") >= 0 ? true : false;
		
		while(in.hasNext()) {
			for(int i = 0;i < delta.length;i++) delta[i] = null;
			dir = "";
			
			s = in.nextLine();
			backup = s;
			
			//Remove https:// or http://
			if(s.substring(0, 8).equals("https://")) s = s.substring(8);//is HTTPS
			else s = s.substring(7); // is HTTP
			
			//split the string via '/'
			delta = s.split("/");
			//System.out.println(delta.length);
			
			if(delta.length == 2 && backup.charAt(backup.length() - 1) == '/') {
				for(int i = 1;i < delta.length;i++) dir += delta[i] + File.separator;
				if(isWin) System.out.println("mkdir " + dir);
				else System.out.println("mkdir -p " + dir);
				System.out.println("curl " + backup + " > " + dir + "index.html");
			}else if(delta.length == 2) System.out.println("curl " + backup + " > " + delta[1]);
			else if(delta.length == 3 && backup.charAt(backup.length() - 1) == '/'){
				for(int i = 1;i < delta.length;i++) dir += delta[i] + File.separator;
				if(isWin) System.out.println("mkdir " + dir);
				else System.out.println("mkdir -p " + dir);
				System.out.println("curl " + backup + " > " + dir + "index.html");
			}else if(delta.length == 3) {
				for(int i = 1;i < delta.length;i++) dir += delta[i] + File.separator;
				if(isWin) System.out.println("mkdir " + dir);
				else System.out.println("mkdir -p " + dir);
				System.out.println("curl " + backup + " > " + dir + delta[1]);
			}
		}
		in.close();
	}
}

看到這樣一堆的if else判斷，我也很無奈嘛，但如果不會java咋辦？沒事，我這邊提供了編譯好的jar檔案，你下載來用，並且安裝Java(JRE)8就好了。

這個~~指令碼~~程式大概就是從命令列讀取輸入的網址，然後幫你按網址建立目錄，並把index.html或者***.html下載到那個目錄。

這個程式我臨時寫的，所以有一些侷限：網址中最多隻可有兩層目錄（比如https://4o5.xyz/archives/86/），需要更多自己寫指令碼吧（捂臉，因為我這邊只有兩層，還有等等bug。

如果要用這個程式，請參照下面步驟：

  1. 開啟cmd/終端，切換到jar檔案所在的目錄。
  2. 輸入java -jar Generator.jar
  3. 然後你看似它不動了，那是因為你還沒有輸入
  4. 輸入一個網址，並按下回車
  5. 然後你會得到幾行命令
  6. 然後複製執行命令即可
  7. 如果想要批量，請先複製好大量網址，一行一個
  8. 然後看第1步，並輸入java -jar Generator.jar > cmd.txt
  9. 然後貼上網址，回車，並按下ctrl + Z再回車，結束程式。
  10. 這時候命令就在cmd.txt裏面了。

因為各自永久連結方式不同，所以你的目錄結構不一定和我一樣。

部署

既然是靜態頁面，部署就非常方便了。

隨便拿個nginx寫個server，或者放oss，或者放github、coding pages都可以。

我這邊大陸訪問直接直連，其餘走cloudflare，然後快取可以設定好久好久（哈哈哈快取一月），順手拿github pages 做備用站點，反正源站被ddos一般直接黑洞了（阿里雲輕量連提醒都不提醒你了），用臺大陸伺服器跑個指令碼遇到不能訪問就切換dns解析，所以被打也無所謂了宕機不超過幾分鐘。

CC攻擊基本上可以說是免疫？因為就單單一個100kb不到的html頁面能被C死，emmmm。

後續操作

儘管我們靜態化了，但是我們要更新文章的話還是得需要原來得部落格框架，所以只要你在釋出新文章得時候要再次執行php+mysql+nginx，然後再次生成html頁面。

畢竟typecho不是靜態部落格。
